Learning module builder notes
=============================

Context
-------

PPOAgentBuilder currently carries both a PolicyBuilder and a LearningModuleBuilder.
A2CAgentBuilder has the same shape. For PPO and A2C, the policy-value
LearningModule is the top-level trainable object, so the builder API should move
toward one learning-module builder that owns policy and value module
configuration together.

The new LearningModuleBuilder2 points in that direction. Before making it part
of the public API, it should be renamed to the final LearningModuleBuilder name,
or otherwise hidden behind a stable public name. The "2" suffix should not leak
into the released API.

Near-term direction
-------------------

Use one learning-module builder inside both PPOAgentBuilder and A2CAgentBuilder.
The agent builders should still own algorithm-specific configuration:

- PPO/A2C params
- PPO/A2C hook builders
- backend selection and device state
- algorithm-specific defaults

The shared learning-module builder should own:

- policy hidden layers
- value hidden layers
- joint vs split policy-value module selection
- optimizer params
- gradient clipping options
- dynamic policy distribution selection from ActionSpaceType

For this release, dynamic policy distribution selection is enough. The builder
can infer categorical policies for discrete action spaces and diagonal Gaussian
policies for continuous action spaces. Explicit distribution selection can be
added later when more distribution types exist.

Deferred issue: generic policy flexibility
------------------------------------------

The Burn PolicyValueModule is generic over the policy type. That is a useful
extension point because callers can build a custom policy and still wrap it in a
policy-value learning module.

The current high-level LearningModuleBuilder2-style API builds the policy
internally and returns a concrete PolicyKind-based module. That is simpler for
PPO/A2C builders, but it narrows the public extension point compared with the
lower-level Burn module API.

This should be revisited before stabilizing the public API:

- Decide whether r2l-api should expose only the ergonomic dynamic builder.
- Decide whether custom Burn policies should remain supported through a separate
  lower-level builder or constructor path.
- Check whether Candle should gain a comparable extension point, or document
  that Candle is intentionally less flexible for now.
- Avoid designing the high-level PPO/A2C builder in a way that prevents adding a
  custom-policy path later.

Naming cleanup
--------------

The current A2C type aliases named A2CBurnLearningModuleBuilder and
A2CCandleLearningModuleBuilder alias A2CAgentBuilder specializations. That naming
is confusing because they are agent builders, not learning-module builders.

When touching this area, rename those aliases toward agent-builder terminology,
or remove them if the clearer A2CAgentBuilder backend specializations are enough.
